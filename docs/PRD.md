# 电商独立站项目立项书与产品需求文档

版本：V1.0
项目类型：单商户 B2C 实物商品电商独立站
开发方式：ChatGPT 负责需求、架构和问题分析，Codex 负责分阶段编码、自测和修复，项目负责人负责本地运行与业务验收。

---

## 一、项目目标

建设一个由商家独立运营的电商网站，实现：

商品展示 → 加入购物车 → 提交订单 → 在线支付 → 商家发货 → 用户查询物流 → 售后处理

网站需要同时包含用户购物前台和商家运营后台，支持手机端访问，并预留支付宝、微信支付、PayPal、Stripe 等支付渠道的扩展能力。

项目不建设多商户入驻、商家分账、直播带货、社交拼团等平台型功能。

---

## 二、项目开发原则

本项目按照以下流程推进：

想法提出
→ ChatGPT 梳理需求
→ ChatGPT 输出立项书与 PRD
→ Codex 分阶段开发并执行自动化测试
→ 项目负责人本地运行验收
→ 将报错、截图、日志和功能偏差交给 ChatGPT
→ ChatGPT 判断根因并输出修复任务
→ Codex 修复和回归测试
→ Git 提交稳定版本

每个开发阶段必须满足三个条件后才能进入下一阶段：

1. 核心功能可以在本地完整运行。
2. 自动化测试和基础构建通过。
3. 已提交 Git，能够恢复到该阶段的稳定版本。

## 三、用户角色

### 1. 游客

可以浏览商品、搜索商品、加入购物车，并直接下单。游客下单时需要填写邮箱、手机号和收货地址。

### 2. 注册用户

除游客功能外，可以管理收货地址、查看订单、收藏商品、申请退款退货、使用积分和优惠券。

### 3. 运营人员

可以管理商品、订单、库存、优惠活动、物流、售后和用户信息。

### 4. 超级管理员

拥有全部后台权限，可以管理运营账号、支付配置、系统配置、操作日志和数据备份。

## 四、核心业务流程

### 1. 用户购物流程

访问网站
→ 浏览或搜索商品
→ 选择商品规格和数量
→ 加入购物车
→ 进入结算页
→ 填写收货地址
→ 选择配送方式
→ 使用优惠码
→ 创建待支付订单
→ 完成支付
→ 库存正式扣减
→ 商家发货
→ 用户确认收货
→ 订单完成

### 2. 游客下单流程

游客选择商品
→ 填写邮箱、手机号和收货地址
→ 创建订单
→ 完成支付
→ 通过订单号和邮箱查询订单

游客完成订单后，可以注册账号并将该订单绑定到新账号。

### 3. 订单售后流程

用户提交售后申请
→ 选择退款或退货退款
→ 填写原因并上传凭证
→ 后台审核
→ 用户寄回商品
→ 商家确认收货
→ 原支付渠道退款
→ 更新订单和库存状态

## 八、支付系统设计

支付模块必须采用支付适配器设计，业务订单不能直接依赖某一家支付服务。

统一接口至少包含：

* 创建支付
* 查询支付状态
* 处理支付回调
* 关闭支付
* 发起退款
* 查询退款状态
* 验证回调签名

支付记录和业务订单分开保存。

一个订单可以出现多次支付尝试，但只能有一次有效支付成功记录。

支付回调必须满足：

1. 验证支付平台签名。
2. 验证支付金额和币种。
3. 验证订单状态。
4. 使用幂等机制，避免重复回调造成重复扣库存。
5. 先保存支付结果，再更新订单状态。
6. 保存原始回调信息和处理日志。

信用卡信息不得保存在本系统数据库。Stripe 等信用卡支付应使用支付平台提供的托管页面或令牌化组件。

## 九、订单与库存规则

### 1. 订单编号

订单编号需要全局唯一，不能使用连续自增 ID 直接暴露订单数量。

建议格式：

`20260718-8F3K92D7`

数据库内部仍使用 UUID 或自增主键，订单编号仅用于用户查询和展示。

### 2. 订单快照

下单时必须保存商品快照，包括：

* 商品名称
* 商品图片
* SKU 名称
* 规格信息
* 商品单价
* 商品数量
* 优惠金额

商品后续改名、改价或删除，不得影响历史订单。

### 3. 库存扣减

推荐流程：

创建订单
→ 数据库事务检查库存
→ 锁定库存
→ 创建待付款订单
→ 支付成功
→ 锁定库存转为实际扣减
→ 支付超时
→ 释放锁定库存

所有库存变更必须生成库存流水。

### 4. 防止超卖

库存检查与库存更新必须在同一个数据库事务内完成，使用条件更新或行锁，不能先查询库存再单独执行普通更新。

## 十二、技术方案

推荐采用 TypeScript 全栈方案，降低 Codex 在多语言项目中的上下文切换成本。

### 项目结构

```text
ecommerce-site/
├─ apps/
│  ├─ storefront/        用户购物前台
│  ├─ admin/             商家运营后台
│  └─ api/               后端 API
├─ packages/
│  ├─ database/          数据库模型和迁移
│  ├─ shared/            公共类型和工具
│  ├─ ui/                公共 UI 组件
│  └─ config/            公共配置
├─ docker/
├─ docs/
├─ tests/
├─ .env.example
├─ docker-compose.yml
├─ README.md
└─ package.json
```

### 推荐技术栈

* 前台：Next.js、TypeScript
* 后台：Next.js、TypeScript
* API：NestJS
* 数据库：PostgreSQL
* ORM：Prisma
* 缓存和任务队列：Redis
* 文件存储：本地开发使用 MinIO，生产环境使用 S3、OSS 或 COS
* 测试：Vitest 或 Jest、Playwright
* 接口文档：OpenAPI
* 本地环境：Docker Compose
* 代码管理：Git 和 GitHub

第一阶段不建议引入 Elasticsearch、微服务、Kubernetes 等复杂基础设施。

* 全站 HTTPS
* 密码哈希存储

* 登录

* 优惠码验证
* 创建订单
* 发起支付

需要记录异常 IP、设备标识、用户账号和请求频率。

### 3. 数据备份

* PostgreSQL 每日备份

## 十四、非功能要求

### 性能

* 首页和商品列表页应支持缓存
* 商品图片需要压缩和懒加载
* 普通页面首屏应在合理网络环境下快速可用

* 后台大数据导出采用异步任务

* 支持手机、平板和桌面端
* 订单支付失败后可以重新支付
* 重复提交订单请求不能创建多份相同订单
* 网络中断后用户可以重新查询订单状态
* 后台危险操作需要二次确认

### 可维护性

* 所有环境变量记录在 `.env.example`
* 数据库变更通过 Prisma Migration
* API 返回格式统一
* 错误码统一管理
* 关键业务代码需要单元测试
* 所有阶段更新 README

## 十七、本地验收规则

每个阶段至少检查：

1. 正常流程。
2. 输入错误。
3. 重复提交。
4. 库存不足。
5. 商品下架。
6. 网络中断。
7. 未登录访问。
8. 越权访问。
9. 手机端页面。

```text

1. 登录测试账号

【预期结果】
只创建一个订单。

创建了两个订单。

【错误信息】
粘贴完整控制台、服务端和数据库日志。

Windows 11
Node.js 版本

浏览器版本
当前 Git commit
```
